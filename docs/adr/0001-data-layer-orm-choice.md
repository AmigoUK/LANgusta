# ADR-0001: Data layer — raw `sqlite3` + thin DAL

- **Status:** Accepted
- **Date:** 2026-04-17
- **Deciders:** maintainer, synthesized from 3-lens council (Pragmatist / Architect / Ecosystem)
- **Supersedes:** —
- **Superseded by:** —

## Context

Spec `docs/specs/02-tech-stack-and-architecture.md §3` defers the ORM decision and recommends raw SQL + a thin data-access layer, with SQLModel only if complexity grows past ~15 tables. `§10` requires `core/` to stay stdlib-only (pure unit-testability). `§15` caps direct runtime dependencies at ≤15.

v1 schema is ~10–15 tables (assets, mac_addresses, timeline_entries, proposed_changes, monitoring_checks, check_results, credentials, scans, tags, asset_tags, config, migrations, review_queue). Modest write volume (~4 writes/sec steady-state, burstier during scans). Append-dominant (timeline, check_results, scan_records grow forever).

## Options considered

### Option A — Raw `sqlite3` stdlib + thin hand-written DAL

Stdlib `sqlite3` with a `langusta/db/queries.py`-shaped module of explicit functions (`get_asset_by_id`, `insert_timeline_entry`). Hand-rolled migration runner keyed off `PRAGMA user_version`.

- Wins: zero new dependencies, greppable SQL, simplest debugging at 2 a.m., cleanest `core/` boundary, no ORM types leaking into domain objects.
- Losses: manual row→dataclass mapping, no autogenerated migrations, DAL sprawl risk if undisciplined.

### Option B — SQLModel

Tiangolo's SQLAlchemy-backed layer. Typed models via Pydantic.

- Wins: typed queries, shares models with Pydantic validation.
- Losses: thin maintainer pool, Pydantic v1→v2 transition still echoes downstream, weak async+SQLite story, ORM types leak through unless carefully segregated.

### Option C — Full SQLAlchemy 2.0 + Alembic

The Python DB standard. Same maintainer org for Alembic.

- Wins: healthiest ecosystem, autogenerate migrations, large community, adoption in comparable tools (NetBox, Paperless-ngx).
- Losses: two dependencies against a ≤15 budget, significant learning surface, ORM types will leak into `core/` absent discipline.

## Decision

**Option A — raw `sqlite3` + thin hand-written DAL**, with a ~100-line migration runner driving `PRAGMA user_version`. No ORM in v1. If v1.5+ schema complexity or contributor friction genuinely exceeds the hand-rolled DAL's carrying capacity, revisit with SQLModel (not SQLAlchemy) as the first step up.

The tipping consideration: LANgusta's invariant that `core/` is stdlib-only means any ORM inevitably either (a) gets walled off into `db/` and adds zero value over hand-written SQL, or (b) leaks types across the boundary and undoes the invariant. Neither is a win, and the zero-dep option is the one without that dilemma.

## Consequences

### Positive

- Zero data-layer dependencies. Install size stays small, `uv tool install langusta` stays instant.
- `core/` remains trivially unit-testable against stdlib only.
- Every query is greppable plain text; operators can paste SQL directly into `sqlite3 langusta.db` to reproduce TUI behaviour.
- Migration story stays under our control — see ADR-0005.

### Negative

- DAL can sprawl into a 2000-line grab-bag. **Mitigation:** one DAL module per aggregate (`db/assets.py`, `db/timeline.py`, `db/scans.py`). Enforced in review.
- No autogenerated migrations. Every schema change is a numbered SQL file by hand.
- Immutable-timeline invariant must be enforced in SQL (triggers or insert-only DAL functions) rather than by an ORM event hook.

### Follow-up work

- Define DAL module conventions in `CONTRIBUTING.md` before the first `db/*.py` file lands.
- Implement `db/connection.py` with a single `connect()` helper that sets WAL, `synchronous=NORMAL`, `foreign_keys=ON`, `busy_timeout=5000`, `temp_store=MEMORY` (spec §3) — never rely on call-site pragmas.
- Add a CI lint that flags raw SQL strings outside `db/` (keep the boundary honest).

## Dissent / unresolved concerns

**Ecosystem lens recommended Option C (SQLAlchemy + Alembic)**, arguing that SQLAlchemy is the healthiest Python DB library by a wide margin (weekly commits, MIT license, same-org Alembic, 2.0 API stable since 2023, overwhelming adoption in comparable self-hosted tools), and that a 3-year-horizon project with 1–2 maintainers should bet on the healthiest ecosystem rather than hand-roll. The counter-argument carried the day on the `core/` purity constraint, but the ecosystem health concern is real: **revisit this ADR if schema passes 15 tables, a second maintainer joins, or hand-rolled migrations start producing bugs at a measurable rate**.
